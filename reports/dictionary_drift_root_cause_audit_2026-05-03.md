# Dictionary Drift Root Cause Audit (2026-05-03)

## Executive Summary

The remaining dictionary drift is not one uniform problem. It has three root
causes:

- Non-contiguous `entry_no` values are generated by merge sequencing in
  `scripts/ops/build_dictionary_inventory.py`.
- Row-level `metadata.hanja` drift comes from merging same-POS homonyms into one
  atom while the old schema stored Hanja only at row level.
- Gloss contamination is mostly legacy rank-collision residue that was already
  generated into staging and inventory artifacts before the current rank guard.

Do not run a broad automatic renumbering pass yet. It would make contaminated
records look cleaner while preserving wrong glosses. Generator fixes and
targeted data repairs should happen first.

## Dictionary Build Data Flow

1. `content_v2/staging/dictionary/main/**/batch_*.jsonl`
2. `content_v2/staging/dictionary/extensions/**/batch_*.jsonl`
3. `content_v2/staging/dictionary/normalized_ranked/*.staging.jsonl`
4. `scripts/ops/build_dictionary_inventory.py`
5. `content_v2/inventory/dictionary/<version>/*.jsonl`
6. `content_v2/core/dictionary/*` and `content_v2/i18n/zh_tw/dictionary/*`

The builder groups staging rows by atom id, merges unique glosses into
`definitions.zh_tw[]`, assigns `entry_no`, writes inventory shards, then refreshes
the root dictionary manifest and release layers.

## Responsible Scripts / Workflows

- `scripts/ops/build_dictionary_inventory.py`
  - Assigns `entry_no`.
  - Merges repeated staging rows for the same atom.
  - Copies row-level Hanja into `metadata.hanja`.
  - Records `rank_collisions` but should not merge collision rows into the
    canonical rank source.
- `scripts/ops/normalize_dictionary_gloss_fill_workflow_outputs.py`
  - Produces gloss-fill staging outputs.
- `scripts/ops/enrich_dictionary_normalized_with_rank.py`
  - Adds rank cleanup metadata to normalized outputs.
- `scripts/ops/audit_dictionary_regressions.py`
  - Data-driven guard against known forbidden gloss regressions.

## Root Cause Hypotheses

| Rank | Root cause | Evidence | Confidence |
|------|------------|----------|------------|
| 1 | `entry_no` assigned from `len(source_refs) + 1` instead of existing entry sequence | `build_dictionary_inventory.py` assigned new senses based on source ref count, so duplicate source mentions could create gaps like `1, 3` | High |
| 2 | Legacy rank-collision propagation polluted staging and inventory | `밤`, `명단`, `문구` final records point to staging rows whose actual atom differs from the rank collision source | High |
| 3 | Row-level Hanja schema cannot represent same-POS homonyms safely | `취하다` style atoms share one atom id but require different Hanja per entry | High |
| 4 | Gloss fill and merge dedup are gloss-string based, not lexical-entry based | `예`, `타다` collect multiple source glosses without entry-level identity | Medium |

## Fix Strategy

### Generator fixes

- Stop assigning new `entry_no` from `len(source_refs) + 1`.
- Assign new merged source rows to the next existing `entry_no`.
- Keep staging path resolution local to the passed `staging_dir` so tests do not
  accidentally read production staging folders.

### One-time data repair

- Repair known contaminated records such as `밤`, `명단`, and `문구` with exact
  JSONL replacements after second-pass review.
- Repair row-level Hanja drift only where definition-level Hanja is explicit.

### Validation guards

- Keep `audit_dictionary_regressions.py` green for existing fixtures.
- Add new forbidden-gloss fixtures only after the corresponding polluted records
  are repaired, otherwise the current manifest will intentionally fail.
- Add entry-number regression tests around `build_dictionary_inventory.py`.

### Manual review only

- Do not infer native/loanword status from translated gloss.
- Do not split atoms into `#1/#2` IDs without a separate migration.
- Do not bulk-renumber existing inventory before corrupted glosses are removed.
