# 平語 (Banmal) 支援能力評估與擴充計畫

## 1. 評估結果：目前可以處理平語嗎？

**短答案：不行。目前的引擎對平語（Banmal）的解析能力近乎為零。**

我撰寫了一個測試腳本，直接將常見的平語輸入目前的 `RulesEngine`，結果如下：

| 測試詞彙 | 預期解析 (V5標準) | 實際引擎輸出 | 錯誤原因 |
| :--- | :--- | :--- | :--- |
| **먹어** | `ko:v:먹다` + `ko:e:어` | `ko:n:먹어` (Fallback) | 缺乏 `-어` 的語尾規則 |
| **해** | `ko:v:하다` + `ko:e:아` | `ko:n:해` (Fallback) | 缺乏 `하다` 變成 `해` 的平語規則 |
| **진짜야** | `ko:n:진짜` + `ko:cop:이다` + `ko:e:야` | `ko:n:진짜야` (Fallback) | 缺乏 `(이)야` 的繫詞規則 |
| **했어** | `ko:v:하다` + `ko:e:았` + `ko:e:어` | `ko:n:했어` (Fallback) | 過去式 `-었/았` 後未接 `어요` |
| **아니야** | `ko:adj:아니다` + `ko:e:야` | `ko:n:아니야` (Fallback) | 缺乏不規則形容詞的平語規則 |

> [!WARNING]
> 因為 A1 級別的課程內容全部都是敬語（-아요/어요, -습니다），目前的 Rules Engine 所有的正則表達式幾乎都是以 `요$` 或 `다$` 結尾。一旦遇到沒有 `요` 的平語，引擎會全面崩潰，觸發 `dictionary_fallback` 把整句話當成一個未知的「名詞 (`ko:n`)」。

---

## 2. 為什麼直接把「요」拔掉不能解決問題？

我們不能單純寫一個腳本說「如果字尾沒有요，就當作平語」。在韓文形態學中，平語的變化有其複雜性：

1. **繫詞 (Copula) 的變化**
   - 敬語：`이다` -> `이에요 / 예요` (ieyo / yeyo)
   - 平語：`이다` -> `이야 / 야` (iya / ya) （例如：학생이야, 진짜야）
2. **疑問句語尾**
   - 平語特有：`-니`, `-냐`, `-어/아`
3. **命令/勸誘句**
   - 平語特有：`-자` (가자), `-라/어라` (먹어라)

## 3. 未來擴充實作計畫 (Implementation Plan)

為了解決這個問題並支援未來的 A2/B1 課程（將會出現大量平語），我們需要在 Rules Engine 中擴充平語專屬的語法樹。以下是建議的實作計畫：

### 階段一：建立平語專屬規則檔 (New Rule Files)

新建兩個規則檔，以確保與現有敬語不衝突，並設定具體的優先權：

1. `engine/rules/31_verb_endings_banmal.json`
   - 涵蓋動詞與形容詞的半語尾：`-어/아`, `-했어`, `-자`, `-니`, `-냐`, `-지` 等。
2. `engine/rules/41_copula_banmal.json`
   - 涵蓋名詞後的平語繫詞：`-(이)야`, `-(이)었어/였어`。

### 階段二：優先權 (Priority) 防禦設計
>
> [!IMPORTANT]
> 平語的語尾通常很短（如單一個 `어`），很容易不小心把其他正常詞彙切壞（例如名詞「상어」(鯊魚) 被切成「상+어」）。
因此，所有平語規則必須設定 `dictionary_required: true`，強制要求切出來的詞幹必須存在於字典中才能成立。

### 階段三：自動化迴歸測試 (Regression Tests)

擴充 `test_rules_engine.py`，加入平語 vs 敬語 的防碰撞測試：

- 測試輸入 `먹어요` 必須正確解析為 `먹다 + 어요`
- 測試輸入 `먹어` 必須正確解析為 `먹다 + 어`
- 測試名詞如 `단어` 不會被誤切為 `단 + 어`
