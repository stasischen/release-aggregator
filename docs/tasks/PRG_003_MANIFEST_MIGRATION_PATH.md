# PRG-003: Manifest & Catalog Migration Path

## Current Roles (Legacy)

### 1. `manifest.json` (Asset Release Truth)
- **Role**: The authoritative index of what is actually deployed in the production environment. It contains exact paths and lesson IDs that the app's asset loader relies on.
- **Action**: Used as the **Primary Seed** for the initial Release Manifest to ensure no existing production content is lost.

### 2. `lesson_catalog.json` (Metadata Supplement)
- **Role**: Defines the unit/lesson hierarchy, titles, and tags for the UI.
- **Action**: Used as the **Secondary Seed** to enrich the Release Manifest with instructional metadata (units, order, tags).

## The Transitional Flow

| Stage | New Release Manifest | `manifest.json` | `lesson_catalog.json` |
| :--- | :--- | :--- | :--- |
| **Phase 0 (Today)** | Non-existent | Authority (Assets) | Authority (Metadata) |
| **Phase 1 (Seeding)** | **Join** (Manifest + Catalog) | Manual/Auto Sync | Manual/Auto Sync |
| **Phase 2 (Cutover)** | **Single Source of Truth** | **Generated Artifact** | **Generated Artifact** |

## Migration Strategy

### 1. Initial Seeding (The Join)
The initial `prd.release_manifest.json` will be generated by joining two sources:
- **Keys**: All `level_id` (lesson IDs) found in `assets/content/production/manifest.json`.
- **Values**: Metadata (unit mappings, titles, tags) found in `assets/content/production/lesson_catalog.json`.
- **Logic**: Any lesson present in `manifest.json` is marked as `release_status: "production"`. Lessons in the catalog but NOT in the manifest may be marked as `staging_only` or `draft`.

### 2. Cutover Logic
Once the Release Manifest is established:
- The Production Assembler takes the Release Manifest as input.
- It generates a production-only `manifest.json` containing only the allowed paths.
- It generates a production-only `lesson_catalog.json` containing only the allowed units and metadata.

### 3. Backward Compatibility
- The Flutter app code **DOES NOT** change; it continues to read `manifest.json` and `lesson_catalog.json` from the same paths.
- The only change is the **workflow**: Humans edit the Release Manifest, and the Assembler updates the deployment artifacts.

## Summary for Implementation

- **Authority Shift**: After Phase 2, `manifest.json` and `lesson_catalog.json` in the production folder are considered **derived outputs**. Direct manual edits to these files will be overwritten by the next build.
- **Join Rule**: When seeding, if a lesson exists in `manifest.json` but is missing from `lesson_catalog.json`, it should be flagged for reconciliation (manual metadata completion).
